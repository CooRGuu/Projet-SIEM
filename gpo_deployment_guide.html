📘 Guide d'Implémentation GPO : Déploiement Zero-Touch Wazuh

1. Préparation de l'infrastructure de distribution

Les fichiers de déploiement doivent être hébergés sur un partage réseau sécurisé :

  1. Créer un répertoire : \\SRV-FICHIERS\Deploy\Wazuh\
  2. Y déposer :
    • Deploy-WazuhAgent.ps1
    • wazuh-agent-4.10.4-1.msi
  3. Appliquer les droits NTFS :
PrincipalDroitRaison
Domain AdminsContrôle totalAdministration
Domain ComputersLecture + ExécutionInstallation par SYSTEM
Authenticated UsersLectureVérification

Modifier la ligne 17 du script :

$MsiNetworkSource = "\\SRV-FICHIERS\Deploy\Wazuh\wazuh-agent-4.10.4-1.msi"

2. Configuration de la GPO

  1. Ouvrir gpmc.msc (Gestion de stratégie de groupe).
  2. Créer une GPO nommée SEC_DEP_WazuhAgent → lier à l'OU cible.
  3. Éditer : Configuration ordinateur → Stratégies → Paramètres Windows → Scripts (démarrage).
  4. Double-cliquer Démarrage → onglet Scripts PowerShell.
  5. Ajouter : \\SRV-FICHIERS\Deploy\Wazuh\Deploy-WazuhAgent.ps1
  6. Sélectionner « Exécuter les scripts Windows PowerShell en premier ».
⚠ Important : Assurez-vous que le délai maximal d'exécution des scripts n'est pas inférieur à 600 secondes (nécessaire pour l'attente du réseau Tailscale).

3. Gestion du secret DPAPI

Le défi actuel (contexte Lab)

DPAPI utilise le scope LocalMachine. La clé est unique au matériel physique du poste. Le fichier api_credential.bin ne peut pas être copié depuis un serveur AD (le poste cible ne pourrait pas le déchiffrer).

Déploiement Lab : Un administrateur doit au préalable provisionner le secret localement via Initialize-WazuhDeployCredential.ps1 sur chaque poste.

Option 3 : Cible de Production (Proxy Kerberos)

Pour un déploiement industriel Zero-Touch sans la contrainte DPAPI :

  1. Retirer les credentials du script GPO. Le script s'authentifie via l'identité Kerberos de la machine (POSTE01$@DOMAIN.LOCAL).
  2. Déployer une API Proxy intermédiaire (Python Flask / Go) qui :
    • Reçoit la requête d'enrôlement du poste.
    • Vérifie dans l'AD si la machine est légitime et active.
    • Utilise ses propres credentials administrateurs (cachés côté serveur) pour l'API Wazuh.
    • Renvoie la clé unique chiffrée à l'agent.

4. Stratégie de mise à jour (Life Cycle)

L'architecture gère nativement l'idempotence et les upgrades :

  1. Déposer le nouveau MSI sur le partage réseau.
  2. Mettre à jour $MsiExpectedHash dans le script.
  3. Au prochain redémarrage, le script détecte la différence et lance l'upgrade.
